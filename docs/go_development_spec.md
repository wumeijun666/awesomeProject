# Go 开发规范手册

> **适用范围**：`D:\awesomeProject`
> **版本**：v1.2.0
> **更新日期**：2026-07-02

---

## 1. 适用范围

本规范适用于本仓库内所有 Go 代码。当前项目以 Ent 和 ELK 代码生成为核心，开发时应优先遵循仓库现有结构、生成流程和 README 中约定的开发方式，而不是套用通用分层模板。

冲突优先级如下：

1. 用户当前明确要求
2. 本规范
3. Go 官方惯例

---

## 2. 项目结构

当前仓库的重点目录如下：

- `main.go`：程序入口
- `ent/schema/`：手写实体定义，新增或修改模型优先改这里
- `ent/entc.go`：Ent/ELK 代码生成入口
- `ent/`：Ent 生成代码，禁止手工修改带有 `Code generated` 标记的文件
- `docs/`：项目文档与开发规范

新增模型时，先改 `ent/schema/*.go`，再执行 `go generate ./...` 更新生成代码。

开发链路按以下顺序执行：

1. 设计/修改 `ent schema`
2. 通过 `ent/entc.go` 生成 Ent `client` 和相关代码
3. 基于 ELK 自动生成基础 CRUD 接口代码
4. 在生成代码之外补充自定义业务 `handler` 和 `service`

其中，基础通用 CRUD 优先复用 ELK 生成结果；`handler` 负责接收参数、调用 `service` 并返回 JSON；`service` 负责参数校验后的业务逻辑编排，并按需调用 Ent client 或基础 CRUD 能力。

---

## 3. 命名与风格

- 包名使用简短、小写、无下划线的命名。
- 导出标识符使用 `PascalCase`，未导出标识符使用 `camelCase`。
- 文件名使用小写加下划线，例如 `todo_schema.go`。
- 使用 `gofmt` 统一格式，不手工调整缩进风格。
- 常量优先使用 `camelCase` 风格，例如 `DefaultTimeout`，避免滥用全大写。

---

## 4. 注释要求

- 公开的包、类型、函数、方法必须有注释，且首行应以名称开头。
- 代码生成文件不要求手工补注释。
- 注释只写必要信息，避免重复描述代码本身已经表达清楚的内容。

---

## 5. Ent 开发约定

- 只修改 `ent/schema/` 下的手写 schema 文件。
- 不直接编辑 `ent/` 目录中生成的代码。
- `ent/entc.go` 是本项目生成入口，生成动作以 `go generate ./...` 为准。
- schema 改动后必须重新生成代码，并确认生成的 `client` 与基础 CRUD 接口可用。
- 如果修改涉及字段、索引或约束，优先同步检查迁移影响。

`ELK` 生成的基础 CRUD 接口同样视为生成代码，原则上不手工改动；如果生成结果不满足业务要求，应回到 `schema` 或生成配置层修正，而不是直接改生成文件。

涉及新增实体时，优先遵循 README 中的流程：

```powershell
go run -mod=mod entgo.io/ent/cmd/ent new User
go generate ./...
```

如果只是修改已有 schema，通常不需要重新 `ent new`，直接调整 schema 后执行 `go generate ./...`。

---

## 6. 接口分层约定

- 基础列表、详情、创建、更新、删除接口，优先复用 ELK 生成的 CRUD 代码。
- 自定义业务接口由手写 `handler` 和 `service` 承担，不要把业务逻辑直接塞进生成层。
- `handler` 负责请求参数接收、基础校验、调用 `service`、返回 JSON 响应。
- `service` 负责业务规则、数据编排、错误处理，以及对 Ent client 或 CRUD 接口的调用。
- 生成代码与手写业务代码应分开维护，避免下次生成时覆盖自定义逻辑。

---

## 7. 错误处理

- 任何返回 `error` 的调用都要显式处理，禁止直接忽略。
- 包装错误时优先使用 `%w` 保留调用链。
- 对于可预期的校验失败，返回清晰、可读的错误信息。

---

## 8. 测试与验证

- 新增逻辑优先补充 `*_test.go` 测试。
- 测试函数使用 `TestXxx` 命名。
- 推荐先运行以下命令验证变更：

```powershell
go generate ./...
go test ./...
go build ./...
go vet ./...
```

如本次改动涉及 `ELK` 生成内容，再额外确认相关 CRUD 接口代码已同步更新。

---

## 9. 提交要求

- 提交信息使用简洁的祈使句，可采用 `feat: ...`、`fix: ...`、`docs: ...` 风格。
- 如果变更涉及 schema，提交时要同时包含 schema 文件、重新生成的 `ent/` 代码，以及受影响的 CRUD 生成代码。
- 如果只改文档或规范，保持提交范围尽量小。

---

## 10. 开发提醒

- 开始修改 Go 代码前，先确认目标是否应该落在 `main.go`、`ent/schema/` 还是生成代码之外的新增目录。
- 不要改动生成代码来承载业务逻辑；生成层负责通用能力，业务扩展放在手写 `handler/service`。
- 如果本规范与任务要求冲突，以任务要求为准，并在交付说明里说明差异。
